# Harness Philosophy

## Core Position

The harness should stay small, strict, and reliable.

It should not become a dump of every idea, reference, or draft. It should keep only the materials that materially support the project's operating model.

## Selection Rule

Bring forward only items that are necessary to at least one of these:

- preserving durable project context
- improving day-start or day-end execution quality
- reducing repeated context rebuild cost
- improving consistency, verification, or handoff quality
- producing a reusable standard harness asset

If an item does not materially improve one of those areas, do not accept it into the standard harness.

## Promotion Rule

Selection alone is not enough.

If a selected item needs editing, supplementation, project-specific adaptation, or conflict resolution, that work must be completed before the item is accepted into the standard harness.

The accepted harness corpus must contain only:

- necessary items
- internally consistent items
- corrected items
- verified items
- reusable items

Selected material from `reference\repo_harness_template` must be adapted into this project's simpler operating model before acceptance. Direct transplant is not allowed.

## Rejection Rule

Reject or keep in candidate status any item that is:

- redundant
- vague
- contradictory
- incomplete
- still coupled to an older project's assumptions
- not worth the maintenance cost

## Operational Consequence

The project may keep many candidate materials, but it should keep a smaller accepted corpus.

The accepted corpus is the standard. The candidate corpus is only a work area.
